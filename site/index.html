Le pipeline de développement, de l'idée à la release.

Forge pilote tout le cycle de développement en étapes courtes, validées une à une. Chaque étape produit un artefact markdown qui alimente la suivante — on n'avance jamais sans validation explicite.

02

Le pipeline

Quatre phases dans le temps, trois tracks symétriques selon la nature du changement. Chaque étape produit un artefact (pitch.md, plan.md, review.md…) validé par un ok explicite avant de passer à la suivante.

PHASE 0

Poser le décor — une fois, documents vivants

  • vision

    Cap produit, personas, non-négociables

    → docs/vision.md

  • product-backlog

    Stories priorisées, prêtes à cadrer

    → docs/product-backlog.md

  • stack

    Choix techniques et conventions de code

    → docs/stack.md

↓
PHASE 1

Cadrer — dans un seul des trois tracks

  • Track Feature

    Valeur utilisateur (user-facing)

    1. (feature-interview → brief.md)
    2. feature-pitch → pitch.md
    3. feature-plan → plan.md

  • Track Refacto

    Comportement figé, code restructuré

    1. refactor-plan → plan.md

  • Track Tech

    Perf, résilience, observabilité, sécu

    1. tech-plan → plan.md
↓
PHASE 2

Implémenter — la skill du track choisi

  • feature-implem

    Le plan devient code fonctionnel

    → code + migrations + tests

  • refactor-implem

    Structure revue, comportement intact

    → code restructuré + tests

  • tech-implem

    Brique posée, effet mesuré

    → code + mesures

↓
PHASE 3

Clôturer — commune aux trois tracks

  1. review

    Sécu, régressions, conformité au plan

    → review.md

  2. report

    L'écart entre l'intention et le livré

    → report.md

  3. sync

    La doc rattrape le code réel

    → docs réalignées

  4. commit

    Message conventionnel, rebase strict

    → commit + push

Tout vit dans docs/story/

Compteur global NNN-<f|r|t>-<slug> : le tri lexicographique donne la timeline du projet. Ex : 042-f-checkout-express/

Track fast — bugfix express

Moins de 3 fichiers, pas de migration ni d'impact transverse : on code, QA du stack, puis /forge:commit. Pas de pitch pour un typo.

03

Les skills

Vingt-cinq skills rangées par phase — le track choisi décide desquelles vous appelez en phases 1 et 2. Perdu en cours de route ? /forge:help est le GPS du pipeline.

Phase 0 — Poser le décor

Documents vivants · 4 modes : Création / Enrichir / Éditer / Pivot

/forge:vision
Cadre la vision : problème, audience, valeur, North Star, principes, anti-objectifs → docs/vision.md
/forge:product-backlog
Traduit la vision en domaines, capacités, parcours et backlog priorisé MVP/V2/V3 → docs/product-backlog.md
/forge:stack
Cartographie la stack technique — chaque techno prouvée par un fichier source → docs/stack.md

Phase 1 — Cadrer

Formaliser l'intention · une seule ligne selon le track choisi

/forge:feature-interview
Feature (optionnel) — découvre un besoin flou par interview guidée, ancrée sur le code existant → brief.md
/forge:feature-pitch
Feature — cadre l'idée et challenge l'alignement vision/backlog → pitch.md
/forge:feature-plan
Feature — plan technique : archi, données, contrats, migration, tests → plan.md
/forge:refactor-plan
Refacto — cadrage + tests de caractérisation à poser comme verrou → plan.md
/forge:tech-plan
Tech — cadrage avec métrique cible chiffrée + baseline + kill switch → plan.md

Phase 2 — Implémenter

Étapes courtes, QA du projet après chacune

/forge:feature-implem
Feature — implémentation guidée sous-tâche par sous-tâche, QA continue
/forge:refactor-implem
Refacto — exécution verrou-tests-d'abord, étapes incrémentales réversibles
/forge:tech-implem
Tech — exécution : baseline, kill switch, mesure après chaque étape

Phase 3 — Clôturer

Commune aux trois tracks

/forge:review
Code review du diff : sécu, qualité, conformité au plan, non-régression → review.md
/forge:report
Compte rendu honnête : fait vs prévu, écarts, dettes → report.md
/forge:sync
Réaligne pitch/plan et les docs projet (vision, stack, product-backlog) sur le code livré
/forge:commit
Message Conventional Commits en français (l'intention), commit + push — vient en dernier

Utilitaires

Hors pipeline

/forge:help
Sommaire du pipeline, tracks, skills et artefacts — le GPS
/forge:claude-md
Génère ou met à jour le CLAUDE.md : analyse du codebase + principes comportementaux
/forge:rules
Écrit les règles projet paths-scopées dans .claude/rules/ — chargées seulement sur la zone concernée
/forge:adr
Rédige un Architecture Decision Record MADR léger → docs/adr/NNNN-slug.md
/forge:estimate
Chiffre le temps « tout compris » d'une story à facturer, en heures → estimate.md
/forge:test-scenario
Joue un scénario utilisateur en live via Playwright MCP
/forge:doc-feature
Cartographie une feature existante (legacy) → docs/feature-map/NNN-slug/overview.md
/forge:backfill-metadata
Reconstruit le metadata.json des stories antérieures depuis l'historique git — jamais de date inventée
/forge:release
Tag SemVer annoté + CHANGELOG.md Keep a Changelog + release GitHub

Inventaire complet : plugins/forge/SKILLS.md · Le workflow détecte le stack (Symfony, Sylius…) via composer.json / package.json et charge les bonnes conventions QA, sécu et perf.

04

Forge Board — l'application

Un tableau kanban lecture seule qui scanne les documents produits par le workflow (docs/story/) et projette chaque story en carte le long d'un pipeline visuel — pour répondre en quelques secondes à « où en est le projet X ? ». L'app Symfony vit à la racine du repo, la marketplace sous plugins/.

FRAMEWORK
Symfony 8.0 · PHP 8.5+ · Symfony CLI (proxy HTTPS *.wip)
FRONT
Symfony UX (Stimulus, Turbo, Live Component) · Tailwind 4 · AssetMapper
DONNÉES
SQLite (fichier local, zéro infra) · Messenger (transport Doctrine)
TESTS
PHPUnit 13 (Unit + Functional) · Playwright (E2E)
QUALITÉ
PHPStan level 9 · PHP-CS-Fixer
MCP
symfony-ai-mate · playwright · chrome-devtools

Démarrer

$ make init # deps PHP/JS + DB + fixtures
$ docker compose up -d # Mailpit (capture des e-mails)
$ make serve # → https://forge-board.wip
make init
Installation complète (deps + DB + fixtures)
make serve
Lance le serveur Symfony
make db-reset
Recrée la base from scratch
make migration
Migration depuis le diff d'entités
make phpunit
Tests PHPUnit (Unit + Functional)
make playwright
Tests E2E Playwright
make quality
CS-Fixer + PHPStan + build
make ci
Lint + tests unitaires (reproduit la CI)

Prérequis : PHP 8.5+, Composer, Symfony CLI, Docker (Mailpit), Node.js 22+. Mailpit UI : localhost:8027 · identifiants de test : admin@example.com / password
